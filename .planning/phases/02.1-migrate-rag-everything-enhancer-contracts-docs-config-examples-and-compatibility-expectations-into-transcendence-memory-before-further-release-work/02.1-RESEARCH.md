# Phase 02.1 Research: Migration Compatibility Recovery

**Project:** Transcendence Memory
**Phase:** 02.1
**Researched:** 2026-03-24
**Confidence:** High

## Objective

Answer the urgent planning question:

What does the planner need to know to repair the missing migration from `rag-everything-enhancer` into `transcendence-memory`, without leaking private details into the public repository?

## Core Audit Result

This is not a normal feature gap. It is a migration contract gap.

The original `rag-everything-enhancer` skill defined:
- a frontend/backend split
- a concrete endpoint/auth/container config shape
- explicit frontend acceptance via `/health`, `/search`, `/embed`
- explicit backend deployment/repair expectations
- explicit LanceDB/container namespace semantics
- explicit safety notes about builtin memory remaining enabled

The current `transcendence-memory` project does **not** preserve those source contracts at the skill/docs layer. Instead it presents a new bootstrap-focused skill and a new backend/CLI implementation. That means the planner should not assume “migration already happened, just polish it.” The planner must explicitly repair the migration surface.

## Planning Conclusions

### 1. This phase must be contract-and-docs heavy before code compatibility is attempted

The immediate risk is misrepresentation:
- the project can look like a migration successor
- the published skill can be installed as if it were a drop-in replacement
- but the important source semantics are missing or changed

So the first planning priority is:
- codify the migration contract
- codify preserved/adapted/dropped behaviors
- ship sanitized source-derived docs and config examples

Only then should any code compatibility helpers or CLI aliases be added.

### 2. The source skill had a richer operational surface than the current one

Source skill docs span:
- setup
- architecture
- dataflow
- operations
- safety/vetting
- env snippet
- config example

Current project skill docs span:
- bootstrap
- troubleshooting

Planning implication:
- this phase should likely add at least 4-6 new docs or references
- the work is not just “update README”; it is “recover missing operator knowledge”

### 3. Config example migration should preserve shape, not secrets

The source frontend config shape matters:
- endpoint
- auth header type/name/value
- default container
- `RAG_CONFIG_FILE`
- load script exported env names

The public migration output should preserve:
- field names and conceptual meaning
- example values as placeholders
- usage notes

It must not preserve:
- private endpoint
- real API key
- internal-only path assumptions unless clearly marked as examples or historical context

### 4. Compatibility must be explicit where semantics changed

The current backend routes are not the same as the source skill’s route contract.

Source assumptions included:
- `container` namespace in request bodies
- centralized LanceDB-per-container storage
- deprecated compatibility aliases

Current implementation includes:
- generic `content` / `query` memory routes
- PostgreSQL + pgvector
- handoff and smoke flow around the new runtime

Planning implication:
- Phase 02.1 should not quietly paper over this difference
- add a compatibility doc or contract matrix that says:
  - preserved
  - adapted
  - not migrated

### 5. Builtin memory rule is migration-critical

The source skill repeatedly states:
- builtin memory remains enabled
- RAG is enhancement only

That rule should become:
- visible in migrated docs
- visible in the skill package
- visible in compatibility notes

### 6. A code-level compatibility shim may be needed, but only if it serves the source contract

Potential code follow-up areas this phase may need to plan:
- config example/template generation matching original frontend shape
- CLI helper to emit source-like config example
- compatibility aliases or notes around old endpoint names if those must be preserved

However, code changes should be driven by the source contract, not by abstract “migration” language.

## Recommended Plan Split

The planner should likely split this urgent phase into:

1. **Migration contract baseline**
   - add docs that map source skill semantics to current project
   - define preserved/adapted/dropped behaviors

2. **Sanitized source-derived docs and config examples**
   - setup, architecture, dataflow, operations, safety
   - env snippet
   - config example

3. **Skill package correction**
   - update `transcendence-memory/SKILL.md`
   - add the new references so published skill no longer misrepresents itself

4. **Compatibility checks**
   - add tests that ensure private values are not reintroduced
   - add docs/tests ensuring migrated contract surfaces remain present

## Validation Architecture

Phase 02.1 validation should focus on:
- source-contract presence
- no-secret regression
- published skill references
- explicit compatibility notes

Suggested tests:
- `tests/unit/test_migration_contract_docs.py`
- `tests/unit/test_sanitized_examples.py`
