<domain>
## Phase Boundary

This phase is not about adding a new feature. It corrects a migration failure. The current `transcendence-memory` skill/package/release surface was presented as if it were the successor to `rag-everything-enhancer`, but the essential contracts from the original skill were not actually migrated.

The phase must therefore recover the missing migration surface:
- original frontend/backend split semantics
- sanitized setup, architecture, dataflow, operations, and safety docs
- sanitized config examples and env/load-script expectations
- explicit compatibility notes for preserved vs adapted vs dropped behaviors

It must not reintroduce private endpoints, private keys, or internal-only assumptions into the public repo.

</domain>

<decisions>
## Implementation Decisions

### Migration Scope
- Treat `rag-everything-enhancer` as the source contract for this urgent phase, not as vague inspiration.
- The migration target is not “clone the private skill verbatim”; it is “preserve the important contracts, examples, and operator expectations in sanitized public form.”
- The phase must explicitly recover the parts that are currently missing:
  - setup guidance
  - architecture/dataflow/operator docs
  - config example shape
  - compatibility expectations

### What Must Be Preserved
- `Keep builtin memory` remains a first-class rule from the original skill.
- The original frontend/backend split remains important and should be visible in the migrated docs and compatibility surface.
- The original client-side contract around endpoint + auth header + default container + env/load-script expectations must be represented in sanitized form, even if the final implementation evolves from LanceDB-only centralized service behavior.
- The original acceptance rule that rollout is not complete until `health`, `search`, and `embed` succeed must be preserved in release/operator docs.

### What Must Not Be Copied Blindly
- Private endpoint `https://rag.zweiteng.tk`
- Real API keys from `rag-config.json`
- Internal-only paths and machine-specific deployment details
- Unreviewed private backend implementation assumptions

### Compatibility Expectations
- The project must explicitly state where it preserves old semantics and where it intentionally diverges.
- In particular, the original `container` namespace model, original LanceDB-only centralized storage model, and compatibility aliases like `build-manifest` / `ingest-memory` must be called out directly, not silently dropped.
- If a behavior is not being reimplemented, the docs must say so clearly rather than implying a drop-in replacement.

### Claude's Discretion
- Exact docs directory structure for migrated materials
- Exact compatibility-manifest format for migration notes
- Exact file layout for sanitized config examples and helper scripts/templates

</decisions>
